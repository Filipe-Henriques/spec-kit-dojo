# Generate Dojo Contract

You are an expert QA Engineer at Sybilion. Your task is to read every artifact in the **current spec folder** and generate a **complete** Dojo blackbox test suite — not a smoke check.

**Locating the spec folder.** Spec-kit puts each feature's artifacts under `specs/<NNN-feature-name>/`, where `NNN` is the numeric prefix that matches the current git branch. Determine it like this:

```bash
git rev-parse --abbrev-ref HEAD            # e.g. 001-task-manager
ls specs/$(git rev-parse --abbrev-ref HEAD)
```

Read **every `.md` file** in that folder, plus any subdirectories. Files generated by `/speckit.specify` and `/speckit.plan` that will be present at this point:

- `spec.md` — the feature spec; the **anchor** for `FR-*` / `SC-*` mapping (mandatory).
- `plan.md` — implementation plan; tech stack, endpoint shapes, error handling decisions.
- `quickstart.md` — how the SUT is started; source for `sut_command` and port.
- `research.md` — design decisions, rejected alternatives, constraints worth knowing.
- `data-model.md` — field types, lengths, ranges, enums, uniqueness rules — directly feeds the §7.A worksheet.
- `contracts/` — API contract files (OpenAPI YAML, JSON Schema, etc.) — when present, treat as authoritative for request/response shapes and supersede ambiguous prose.

If `data-model.md` or a file in `contracts/` defines a constraint not mentioned in `spec.md`, the constraint still belongs in the test suite. Read all of them before filling out a single worksheet.

**Not yet available at this point** — do not look for them: `tasks.md` (generated later by `/speckit.tasks`) and any `/speckit.analyse` output (runs even later, and is optional).

**When this runs.** This command fires after `/speckit.plan` and **before** `/speckit.implement`. No application code exists yet — your sole sources of truth are the spec-folder artifacts above. The tests you write here will initially fail; `/speckit.implement` then writes the implementation to satisfy them, and `/speckit.sybilion-dojo.validate` runs after to confirm the suite is green. Treat your tests as the **acceptance contract** for the implementation: if the test would pass against a wrong implementation, it is too loose.

A complete suite catches every observable failure mode the spec describes. Three artifacts in this template force completeness:

- **Pre-flight worksheet** (§7.A) — derives test multipliers from each endpoint.
- **Coverage matrix** (§7.C) — maps every `FR-*` and `SC-*` to ≥1 test before any test is written.
- **Self-audit checklist** (§8) — run before stopping. For every failing item, write the missing test.

If the final suite has fewer tests than `endpoints × applicable categories`, you stopped too early. Go back to §7.

## Dojo Rules & Constraints

### 1. Directory Structure

```
tests/blackbox/
  dojo.yaml                                       ← REQUIRED suite config
  COVERAGE.md                                     ← REQUIRED audit artifact
  .env                                            ← OPTIONAL; only if the SUT or any API needs env config
  .env.local                                      ← OPTIONAL; gitignored secrets — required only for live-mode APIs
  startup.plan                                    ← OPTIONAL; required if SUT makes outbound calls on boot
  eval.md                                         ← OPTIONAL; only if any test uses Evaluate Response
  seed/
    schema.sql                                    ← OPTIONAL; only if the SUT uses PostgreSQL
  test_<resource>_<operation>_<scenario>/
    <name>.plan                                   ← exactly one .plan per directory
    <fixture>.json                                ← fixtures referenced in the plan
    eval.md                                       ← OPTIONAL test-specific eval rules
    dojo.yaml                                     ← OPTIONAL test-level overrides
    seed/
      seed.sql                                    ← OPTIONAL test-specific DB seed
```

- All tests MUST be placed inside `tests/blackbox/`.
- Each test MUST have its own directory starting with `test_`.
- Each test directory MUST contain exactly one `.plan` file.
- Fixture files (`.json`, `.sql`) MUST be placed in the same directory as the `.plan` file.
- **`tests/blackbox/dojo.yaml` MUST always be created.** Without it, Dojo errors with `Suite 'blackbox' not found in workspace 'tests'`.
- **`tests/blackbox/COVERAGE.md` MUST always be created** with the pre-flight worksheets and coverage matrix (see §7).

**Naming convention — encode intent in the directory name:**

Directory names MUST follow `test_<resource>_<operation>_<scenario>/`. A reviewer should be able to scan the suite without opening files.

- `<resource>`: the noun (`task`, `order`, `user`)
- `<operation>`: the action (`create`, `list`, `delete`, `update`, `summarize`)
- `<scenario>`: what is being asserted (`success`, `empty_title`, `not_found`, `idempotent`, `title_over_max`, `upstream_500`)

Examples:
- `test_task_create_success/`
- `test_task_create_empty_title/`
- `test_task_create_title_over_max/`
- `test_task_list_empty/`
- `test_task_list_ordered/`
- `test_task_delete_idempotent/`
- `test_task_summarize_upstream_500/`

### 2. The `dojo.yaml` Suite Configuration

`tests/blackbox/dojo.yaml` is mandatory. Full schema:

```yaml
concurrency: 1
strict_duplicate_expects: true          # enforces uniqueness even at concurrency 1

sut_command: "<shell command to start the SUT>"
sut_base_url: "http://127.0.0.1:<port>"

timeouts:
  perform: 10s                           # SUT HTTP trigger timeout
  expect:  5s                            # outbound Expect fulfilment timeout
  sut_startup: 30s

# Named entry points (optional — see "Entrypoints" below).
entrypoints:
  webhook:
    type: http
    method: POST
    path: /webhook
    body_type: json                      # json | form
    headers:
      X-Api-Key: "test_secret"

# External services Dojo will intercept (HTTP APIs and PostgreSQL).
apis:
  gemini:
    mode: mock                           # mock | live  — see "API Modes" below
    url: "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent"
    timeout: 30s                         # overrides global expect timeout
    headers:
      Authorization: "Bearer $GEMINI_API_KEY"
    default_response:
      code: 200
      body: '{"candidates": [...]}'

  postgres:
    mode: live
    protocol: postgres
    url: "postgres://user:pass@host:5432/db?sslmode=disable"

# Suite-level Postgres seed files; run once at suite startup.
seed:
  - seed/schema.sql
```

**How to determine `sut_command`:**
- Read `quickstart.md` or `README.md` for the exact server-start command.
- Use the project's virtualenv if Python: `.venv/bin/uvicorn <module>:app --host 127.0.0.1 --port <port>`.
- Pick a unique port (e.g. `18000`) to avoid conflicts with the dev server.
- If the app accepts a DB path via env var, point it at a temp path: `TASK_MANAGER_DB=/tmp/dojo_test.db .venv/bin/uvicorn ...`

**APIs section** — only add entries for services Dojo can actually intercept: outbound HTTP APIs (Stripe, Gemini, etc.) and **PostgreSQL** (the only database Dojo supports, via the Postgres wire protocol). Any other database — SQLite, MySQL, MongoDB, Redis, etc. — cannot be intercepted and must not be listed here. Prefer PostgreSQL when persistence is part of the contract you want to test (see §5).

#### 2.1 API modes: `mock` vs `live`

Each API entry MUST set a `mode`.

| Mode | What it does | Use when |
|---|---|---|
| `mock` | Returns the configured `default_response` (or per-`Expect` `Respond:` fixture). No real upstream call. | **Default for contract tests.** Keeps tests deterministic, fast, and free of external credentials. The `Expect` line matches the SUT's outbound request; the response comes from the mock. |
| `live` | Proxies to the real upstream URL. Dojo observes traffic but the upstream provides the response. | Integration validation against the real service (Postgres, or an explicit "does the real Stripe accept our request" check). Requires credentials in `.env.local`. |

Rule of thumb: mock everything except Postgres. Use live mode for one or two explicit integration tests per external API, not the whole suite.

#### 2.2 Test-level `dojo.yaml` overrides

Any test directory can include its own `dojo.yaml` that overlays the suite config. Use for per-test timeouts, auth headers, or alternate entry points.

```yaml
# test_task_summarize_slow/dojo.yaml
apis:
  gemini:
    timeout: 60s            # extend timeout for a deliberately slow LLM test
entrypoints:
  webhook:
    headers:
      X-Api-Key: "test_secret_admin"
```

Only specify differing fields — Dojo deep-merges suite first, then the test overlay.

#### 2.3 Suite-wide and per-test database seeds

- **Suite-wide:** files listed under `seed:` in `dojo.yaml` run once at suite startup (typically schema DDL).
- **Per-test:** any `test_*/seed/*.sql` file runs serially **before that test only**.
- A failing seed aborts the entire suite (even tests that would otherwise pass are reported as failed).
- For state-precondition tests on Postgres, prefer a per-test seed over a long multi-step `Perform` chain.

### 3. Environment Files

Dojo loads `tests/blackbox/.env` (committed) then `tests/blackbox/.env.local` (gitignored, overrides). Variables are injected into both Dojo and the SUT process before startup.

**Auto-injected by Dojo:** for every entry in `apis:`, Dojo sets `API_<NAME>_URL` to the proxy URL. The SUT must read this env var instead of hard-coding the upstream URL.

```bash
# .env — committed defaults; reference proxy URLs Dojo injects
GEMINI_BASE_URL=$API_GEMINI_URL
DATABASE_URL=$API_POSTGRES_URL

# .env.local — gitignored; real secrets for live mode
GEMINI_API_KEY=sk-real-key-here
```

If any API is in `mode: live`, its credentials MUST live in `.env.local` (never `.env`).

### 4. The `.plan` DSL Syntax

The `.plan` file uses a strict `Action -> Target -> Clause` syntax.

- Every plan MUST start with a `Perform` action to trigger the SUT.
- `Expect` lines declare outbound calls the SUT must make to **external APIs or Postgres** (proxied by Dojo).

**Two equivalent `Perform` trigger forms:**

```
# Literal HTTP path — fine for simple cases
Perform -> POST /tasks -> Payload: request.json -> ExpectStatus: "201"

# Named entrypoint — required if the entry has fixed headers / body_type defined in dojo.yaml
Perform -> entrypoints/webhook -> Payload: incoming.json -> ExpectStatus: "200"
```

**Other valid `Perform` lines:**

```
Perform -> GET /tasks -> ExpectStatus: "200"
Perform -> DELETE /tasks/1 -> ExpectStatus: "204"
Perform -> wait -> 500ms
Perform -> postgres -> Query: check_state.sql -> Expect: expected_rows.sql
```

**Valid `Expect` syntax (outbound API calls only):**

```
Expect -> postgres -> Request: insert_user.sql
Expect -> stripe_api -> Request: charge.json -> Respond: charge_success.json
Expect -> gemini  -> Request: prompt.json   -> MaxCalls: "3"
Expect -> gemini  -> Request: prompt.json   -> Evaluate Response
```

**`ExpectStatus` clause** — always include `ExpectStatus` on every `Perform` line. Without it, Dojo fails on any HTTP status ≥ 400, causing false failures on tests that intentionally trigger 4xx responses.

**Multi-step `Perform` — seed state inline:**

A single `.plan` can chain multiple `Perform` lines to set up state before the assertion. Use this for state-transition and idempotency tests when a per-test seed is overkill.

```
# test_task_delete_idempotent/delete_idempotent.plan
Perform -> POST /tasks -> Payload: seed_task.json -> ExpectStatus: "201"
Perform -> DELETE /tasks/1 -> ExpectStatus: "204"
Perform -> DELETE /tasks/1 -> ExpectStatus: "404"
```

```
# test_task_create_then_get/create_then_get.plan
Perform -> POST /tasks -> Payload: new_task.json -> ExpectStatus: "201"
Perform -> wait -> 50ms
Perform -> GET /tasks -> ExpectStatus: "200"
```

`Perform -> wait -> <duration>` inserts a pause. Use it for async write-then-read flows or to space ordered timestamps.

**`MaxCalls` semantics — greedy with lookahead:**

```
Expect -> gemini -> Request: tool_call.json -> MaxCalls: "5"
Expect -> gemini -> Request: final_call.json -> Respond: final_response.json
```

The first expectation consumes up to 5 matches *but advances early* the moment a request matches the next expectation. So 3 tool calls then a final call satisfies both lines (the 3 don't have to reach 5). `MaxCalls` cannot be combined with `Respond:` — it is for live-mode APIs only.

**`Evaluate Response` — for non-deterministic SUT outbound payloads:**

When the SUT generates an outbound request whose body cannot be matched exactly (LLM prompts containing timestamps, idempotency keys, UUIDs), replace exact `Request:` matching with `Evaluate Response`:

```
Expect -> gemini -> Request: prompt.json -> Evaluate Response
```

Dojo sends the SUT's actual outbound payload, plus the merged `eval.md` rules (suite-level + test-level), to an LLM grader. The grader returns pass/fail.

Place rules in `tests/blackbox/eval.md` for suite-wide rules and `test_*/eval.md` for test-specific rules. Prefix a test-level rule line with `+` to **append** to the suite rules; otherwise the test file replaces them.

Example `tests/blackbox/eval.md`:
```markdown
The payload must be a valid JSON object with `contents` and `generationConfig` keys.
`contents[0].parts[0].text` must mention the task title verbatim.
Do not fail due to timestamps, request IDs, or model parameters.
```

Use `Evaluate Response` only when exact match is genuinely impossible — fixture matching is cheaper, faster, and more precise when it works.

**`startup.plan` — outbound calls during SUT boot:**

If the SUT makes outbound calls *while starting* (license check, cache warm-up, schema migration), every test will fail until those calls are captured in `tests/blackbox/startup.plan`. Only `Expect` lines are allowed (no `Perform`); fixtures resolve from the suite directory.

```
# tests/blackbox/startup.plan
Expect -> gemini -> Request: startup_warmup.json -> Respond: startup_warmup_response.json
Expect -> license_server -> Request: license_check.json -> Respond: license_ok.json
```

If the SUT does not make any boot-time outbound calls, omit this file entirely.

### 5. Database Assertions

**PostgreSQL is the only database Dojo can intercept.** Dojo speaks the Postgres wire protocol, so it can observe and assert on the exact SQL the SUT runs against Postgres — giving you a true contract test that validates the persistence side-effect, not just the HTTP response. Every other database (SQLite, MySQL, MongoDB, Redis, DynamoDB, …) is opaque to Dojo and must be validated indirectly through the HTTP surface. Whenever the plan has a choice of database, recommend PostgreSQL so persistence can be enforced end-to-end.

**PostgreSQL (recommended — DB-level assertions):** add to `dojo.yaml`:
```yaml
apis:
  postgres:
    mode: live
    protocol: postgres
    url: "postgres://user:pass@host:5432/db?sslmode=disable"
```

Then in a `.plan`:

```
# Validate a SUT-issued query and seed the response
Expect -> postgres -> Request: insert_task.sql

# Run a verification query yourself
Perform -> postgres -> Query: count_tasks.sql -> Expect: expected_count.sql
```

**Any other database (HTTP-only validation):** do NOT add the database to `apis:` and do NOT write `Expect -> <db> -> ...` lines — Dojo will error with `API '<db>' is not defined`. Validate persistence indirectly:
- After a `POST` that creates a resource, follow with a `GET` to confirm the resource is listed.
- After a `DELETE`, follow with a `GET` to confirm the resource is gone (or returns 404).

### 6. Fixtures

- You MUST generate every fixture file referenced in the `.plan`.
- HTTP request bodies MUST be `.json` files with the exact payload the SUT expects.
- SQL files are used for PostgreSQL assertions (the only DB Dojo can intercept); any other database has no direct DB-assertion path.
- Do not use placeholders. Fixtures represent the exact payload on the wire.

**Fixture realism — boundary values must be exact:**

Negative-test fixtures MUST contain the exact value that triggers the constraint. Vague stand-ins do not exercise the boundary.

| Test category | Fixture value | Wrong |
|---|---|---|
| Empty string | `""` | `"x"`, `" "`, `null` |
| Whitespace-only | `"   "` (3+ spaces) | `" "` (single space — some parsers trim) |
| Over-max length | exactly `N+1` characters where `N` is the spec limit (write the literal string) | `"long"`, `"x" repeated 100 times` |
| Under-min length | exactly `min-1` characters | `""`, `"a"` (unless that *is* `min-1`) |
| Not-found ID | `999999` | `1`, `0`, `999` (may exist) |
| Out-of-range numeric | exactly `max+1` or `min-1` | `100`, `-1` chosen without checking the spec |
| Invalid enum | a string not in the enum (e.g. `"urgent"` when enum is `low|med|high`) | `""`, `"low "` (trailing space — may be trimmed) |
| Wrong type | the wrong JSON type (e.g. `123` where a string is expected) | a string that merely *looks* invalid |
| Ordering test | ≥3 distinct items, created with `wait` between them so timestamps are unambiguous | 2 items, or items created in the same millisecond |

A fixture file should be readable as direct evidence that the test exercises the right boundary.

### 7. The Coverage Engine — How to Produce a Complete Suite

This is the core of the template. Follow A → B → C in order. Do **not** skip ahead to writing `.plan` files.

#### 7.A Pre-flight worksheet (one per endpoint)

For every endpoint in the spec, fill out the following block in `tests/blackbox/COVERAGE.md`:

```
Endpoint: <METHOD> <path>
Implements: FR-XXX, FR-YYY, SC-ZZZ
Required fields:    <name> (<type>, <constraints>)
Optional fields:    <name> (<type>, <constraints>)
Returns:            <status codes and meanings — e.g. 201 created, 400 validation, 404 not found>
Resource-scoped:    yes | no   (operates on /<resource>/{id})
Mutating:           yes | no
Destructive:        yes | no   (DELETE or equivalent)
Outbound APIs:      <list of external APIs the SUT calls for this endpoint, or "none">
Non-deterministic:  yes | no   (SUT sends timestamps/UUIDs/LLM prompts outbound, OR returns LLM output)
Reads-after:        <endpoints that should reflect this mutation>
Unique constraints: <fields if any, else "none">
```

This worksheet is the multiplier. The number of tests for an endpoint scales with the number of fields, constraints, and outbound APIs listed here.

#### 7.B The taxonomy matrix

For each endpoint, generate **one test per applicable cell**:

| Category | Required when | Trigger |
|---|---|---|
| Happy path | always | canonical valid request → expected success status |
| Missing required field | per required field | omit that one required field |
| Empty string | per required string field | field = `""` |
| Whitespace-only | per required string field | field = `"   "` (3+ spaces) |
| Over-max length | per length-bounded string field | field = string of exactly `N+1` chars |
| Under-min length | per length-bounded string field with `min > 0` | field = string of exactly `min-1` chars |
| Wrong type | per typed field (if spec defines types) | field has wrong JSON type |
| Out-of-range numeric | per range-bounded numeric field | value = exactly `min-1` or `max+1` |
| Invalid enum | per enum field | value not in the enum |
| Not found | resource-scoped operations | id = `999999` |
| Empty state | per collection GET | list before any data is created |
| Ordering | per ordered collection | create ≥3 items with `wait` between them; list; expected order in the response |
| State transition | per mutating endpoint | mutate → read-after → confirm visible change |
| Idempotency | per destructive endpoint | repeat the destructive op; second call returns 404, not 500 |
| Conflict | if uniqueness constrained | create duplicate of unique resource |
| Outbound contract | per outbound API call | `Expect -> <api>` matches the SUT's outbound request fixture |
| Non-deterministic outbound | per outbound call with dynamic payload | `Expect -> <api> -> Evaluate Response` + `eval.md` |
| Upstream failure | per outbound API | mock the API to return 5xx; assert the SUT degrades gracefully (e.g. 502, retry, fallback) |

**Skip-with-justification.** If a category does not apply to an endpoint, record it in the coverage matrix as a `(skip)` row with a one-line reason (e.g. `task/conflict — no uniqueness constraint in spec`). Do not silently skip.

#### 7.C The coverage matrix

Before writing any `.plan`, produce this table in `tests/blackbox/COVERAGE.md` mapping every `FR-*` and `SC-*` to ≥1 test:

| Source | Test name | Category |
|---|---|---|
| FR-001 | test_task_create_success | happy path |
| FR-002 | test_task_create_missing_title | validation: missing field |
| FR-002 | test_task_create_empty_title | validation: empty string |
| ... | ... | ... |
| (skip) | task/conflict — no uniqueness constraint in spec | — |

Rules:
- Every `FR-*` from `spec.md` MUST appear in at least one row.
- Every `SC-*` from `spec.md` MUST appear in at least one row.
- Every skipped category MUST appear as a `(skip)` row with a one-line reason.

### 8. Self-Audit Checklist — Run Before Stopping

Agents stop at "looks done." This checklist is what makes "looks done" into "actually done." After writing all tests, work through every item. For each failing item, write the missing test and re-run the checklist from the top.

```
[ ] tests/blackbox/dojo.yaml exists with correct sut_command, sut_base_url, timeouts, and apis
[ ] tests/blackbox/COVERAGE.md exists with all pre-flight worksheets and the coverage matrix
[ ] Every FR-* from spec.md appears in the coverage matrix
[ ] Every SC-* from spec.md appears in the coverage matrix
[ ] Every endpoint has at least one happy-path test
[ ] Every required field has missing / empty / whitespace tests (or a justified skip)
[ ] Every length-bounded string field has an over-max-length test
[ ] Every length-bounded string field with min > 0 has an under-min-length test
[ ] Every range-bounded numeric field has out-of-range tests (min-1 and max+1)
[ ] Every enum field has an invalid-enum test
[ ] Every resource-scoped endpoint has a not-found test (id = 999999)
[ ] Every collection endpoint has an empty-state test
[ ] Every ordered collection has an ordering test (≥3 items, distinct, asserted order)
[ ] Every mutating endpoint has a state-transition test (write → read-after → confirm)
[ ] Every destructive endpoint has an idempotency test (delete twice → 404, not 500)
[ ] Every unique-constrained resource has a conflict test
[ ] Every outbound API call has an outbound-contract test (Expect with fixture)
[ ] Every outbound API has an upstream-failure test (mock 5xx → SUT degrades gracefully)
[ ] Every non-deterministic outbound payload uses Evaluate Response + eval.md
[ ] If the SUT makes outbound calls on boot, tests/blackbox/startup.plan captures them
[ ] Every API in apis: declares mode (mock | live) and an appropriate timeout
[ ] Every live-mode API has credentials in .env.local (never .env)
[ ] Every "MUST" sentence in spec.md has at least one test asserting violation is rejected
[ ] Every .plan has ExpectStatus on every Perform line
[ ] Every fixture referenced in a .plan exists in its directory
[ ] Every negative-test fixture uses exact boundary values (per §6 realism table)
[ ] Every test directory name follows test_<resource>_<operation>_<scenario>
[ ] No two tests share the same name
[ ] No test depends on the leftover state of another test (each plan seeds what it needs)
```

If you think the suite is done, you are wrong. Run this checklist first.

### 9. Worked Example

A task manager feature with four endpoints:
- `POST /tasks` — required: `title` (string, 1-500 chars); optional: `description` (string, ≤2000 chars), `priority` (enum: `low|med|high`).
- `GET /tasks` — returns list, newest-first.
- `DELETE /tasks/{id}` — deletes by id.
- `POST /tasks/{id}/summarize` — calls Gemini to produce a short summary of the task; returns 200 with `{summary: string}`.

**Pre-flight worksheet for `POST /tasks`:**

```
Endpoint: POST /tasks
Implements: FR-001, FR-002, FR-003
Required fields:    title (string, 1-500 chars)
Optional fields:    description (string, <=2000 chars), priority (enum: low|med|high)
Returns:            201 created; 400 validation
Resource-scoped:    no
Mutating:           yes
Destructive:        no
Outbound APIs:      none
Non-deterministic:  no
Reads-after:        GET /tasks must include the new task
Unique constraints: none
```

**Pre-flight worksheet for `POST /tasks/{id}/summarize`:**

```
Endpoint: POST /tasks/{id}/summarize
Implements: FR-007
Required fields:    id (path param, integer)
Optional fields:    none
Returns:            200 with {summary}; 404 if task missing; 502 if Gemini fails
Resource-scoped:    yes
Mutating:           no
Destructive:        no
Outbound APIs:      gemini  (prompt includes task title; mode=mock for the suite)
Non-deterministic:  yes     (SUT prompt embeds task title verbatim; exact-match fixture works,
                             but if the prompt also includes a timestamp use Evaluate Response)
Reads-after:        n/a
Unique constraints: none
```
